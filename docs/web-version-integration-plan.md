# Web Version Integration Plan

Last updated: 2026-05-06 Asia/Taipei

## Product Direction

This project should become the mobile web version of the existing LINE bot.

It is not a replacement for the LINE bot, and it should not invent a new card language. The existing LINE Flex card format and strategy content remain canonical.

The web version should provide:

- Everything the original LINE card provides.
- A deeper stock detail page.
- K-bar chart and cached K-bar history.
- Google Sheets scan results, watchlist, whitelist, and performance data.
- Warrant Top 3 and deeper warrant candidates.
- RS ranking, funnel stats, stale data warnings, cache status, and LINE quota/status.

## Important Constraint

Do not put secrets in this static web repo.

The following must stay server-side:

- `FINMIND_TOKEN`
- `LINE_TOKEN`
- `LINE_SECRET`
- `GOOGLE_CREDENTIALS`
- Google Sheets write access
- LINE push/reply APIs
- Any admin-only operation

Because of this, GitHub Pages alone cannot directly run the full bot functionality. It can only display static files or call public/read-only APIs.

## Recommended Architecture

Use the existing `JustinFang1019/justin-trading-bot` backend as the source of truth.

```text
LINE Bot / Flask backend
  - patterns.py
  - notify.py
  - query.py
  - watchlist.py
  - warrant.py
  - disk K-bar cache
  - Google Sheets
        |
        | read-only JSON APIs
        v
Mobile Web / LIFF viewer
  - index.html or future frontend app
  - renders original card payload
  - shows deeper detail pages
```

## API Surface Needed

Phase 1 should be read-only.

Suggested endpoints:

- `GET /api/web/scan-results`
  - Today's scan results from Google Sheets.
  - Must include the same fields used by original cards.

- `GET /api/web/stock/<sid>`
  - Stock detail, strategy fields, K/D/gap, A/B/C layers, stop, pressure, performance.

- `GET /api/web/stock/<sid>/kbar`
  - Uses existing disk K-bar cache and existing refresh behavior.
  - Should not fetch expensive data from the browser.

- `GET /api/web/stock/<sid>/card`
  - Returns canonical Flex JSON or a canonical card payload generated by `notify.py`.
  - This is the key endpoint for preserving original card format.

- `GET /api/web/stock/<sid>/warrants`
  - Uses existing warrant cache and Top 3 logic.

- `GET /api/web/rs`
  - Uses existing RS baseline.

- `GET /api/web/funnel`
  - Uses existing funnel stats.

- `GET /api/web/status`
  - K-bar freshness, Sheets status, warrant cache, quota, scheduler status.

## Rendering Original Cards

Best option:

1. Refactor the original bot so `notify.py` can generate a canonical card payload without sending it.
2. The web API returns that payload.
3. The web app renders that payload with a small Flex-compatible renderer, or displays it side-by-side with a web detail panel.

Do not manually recreate the LINE card in `index.html` long-term. That will drift and become wrong.

## Suggested Implementation Order

1. In `justin-trading-bot`, add read-only API routes.
2. Add a card payload export path from `notify.py`.
3. Add CORS or host the web app from the same Flask backend.
4. In `justin-trading-bot-web`, replace prototype arrays with API calls.
5. Add LIFF only after read-only pages are correct.
6. Add write actions later with authentication:
   - add/remove watchlist
   - admin rescan
   - alert settings
   - push/re-push controls

## Current Repo Role

`justin-trading-bot-web` currently remains a static prototype and design target. It can be used to design the mobile experience, but the real full-function version needs backend APIs from the original bot.
